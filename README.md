# FileSync Notes

一个面向 Windows 的轻量桌面同步工具，用来把统一笔记目录中的文档，单向同步到项目里的 `README`、教程文档或文档目录。

适合这样的工作流：

- 笔记统一维护在 Obsidian 或普通文件夹里
- 项目仓库里只保留最终需要提交的文档副本
- 修改源笔记后，希望目标文件自动同步，避免手动复制

## 功能概览

- 单向同步规则
  - `文件 -> 文件`
  - `文件夹 -> 文件夹`
- 可选双向同步规则
  - `文件 <-> 文件`
  - `文件夹 <-> 文件夹`
- 源路径必须存在，目标路径允许不存在
- 首次同步自动创建目标目录和目标文件
- 自动同步
  - 文件监听
  - 轮询兜底
- 手动同步
  - 单条规则立即同步
  - 全部规则立即同步
- 备份机制
  - 仅在覆盖已有目标文件前创建备份
  - 备份写入目标同级 `.back` 目录
  - 支持按保留天数自动清理
- 托盘常驻
  - 开机自启
  - 关闭窗口后隐藏到托盘
- 调试辅助
  - 同步历史
  - 运行日志
- 异常规则隔离
  - 源路径失效、源类型错误或监听初始化失败时，只标记规则异常
  - 不会因为单条规则异常导致软件无法启动

## 技术栈

- 前端：React 19 + TypeScript + Vite
- 桌面端：Tauri 2
- 后端：Rust

## 目录结构

```text
E:\code\other\FileSync
├─ public/                  静态资源
├─ src/                     React 前端源码
├─ src-tauri/               Tauri / Rust 桌面端源码
│  ├─ icons/                安装包和托盘图标资源
│  ├─ src/                  Rust 业务逻辑
│  └─ tauri.conf.json       Tauri 配置
├─ package.json             前端与 Tauri 脚本
├─ README.md
└─ .gitignore
```

说明：

- `dist/` 是前端构建产物，不需要提交到 GitHub
- `src-tauri/target/` 是 Rust/Tauri 构建产物，不需要提交到 GitHub
- `dist-portable/` 或其他本地测试目录也不需要提交

## 开发

安装依赖：

```powershell
npm install
```

启动开发环境：

```powershell
npm run tauri:dev
```

仅构建前端：

```powershell
npm run build
```

## 打包

调试构建：

```powershell
npm run tauri:build:debug
```

Release 安装包：

```powershell
npm run tauri:build:release
```

调试便携版（仅生成 exe，不生成安装包）：

```powershell
npm run tauri:build:portable:debug
```

Release 便携版（仅生成 exe，不生成安装包）：

```powershell
npm run tauri:build:portable:release
```

默认输出位置：

- 调试可执行文件：
  - `src-tauri\target\debug\filesync-notes.exe`
- Release 可执行文件：
  - `src-tauri\target\release\filesync-notes.exe`
- Release 安装包：
  - `src-tauri\target\release\bundle\nsis\`
  - `src-tauri\target\release\bundle\msi\`

说明：

- `tauri build --no-bundle` 仍然会生成完整的前端资源并编译桌面端，只是跳过 NSIS/MSI 安装包步骤
- 便携版适合本地开发、自测和临时分发；如果需要开机自启、开始菜单入口和稳定升级路径，仍建议使用安装版
- 开机自启依赖“当前 exe 的绝对路径”，因此不建议在 `npm run tauri:dev` 模式下验证该功能
- 开启开机自启后，应用会在登录 Windows 时以托盘后台模式启动；如果你移动了便携版 exe，重新打开应用后会自动校准启动项路径

## 使用说明

### 文件规则

适用于：

- 单个 Markdown 文档同步到项目中的单个目标文件

示例：

- 源：`E:\notes\UE\README.md`
- 目标：`E:\project\README.md`

注意：

- 目标路径是“目标文件完整路径”
- 如果你先选了目标目录，程序会自动补上源文件名

### 文件夹规则

适用于：

- 一整个文档目录同步到项目文档目录

示例：

- 源：`E:\notes\docs`
- 目标：`E:\project\docs`

### 备份

- 覆盖已有目标文件前，会在目标同级 `.back` 目录生成备份
- 如果本次目标内容与源内容一致，则不会覆盖，也不会生成备份

## 同步策略

当前实现是固定的单向同步策略，核心规则如下：

- 同步方向
  - 只支持 `源 -> 目标`
  - 不会把目标侧的修改反向写回源文件
- 文件规则
  - 源必须是单个文件，目标必须是单个文件路径
  - 如果目标不存在，会自动创建父目录并写入目标文件
  - 如果目标已存在且内容一致，则跳过
  - 如果目标已存在且内容不同，则先备份再覆盖
- 双向文件规则
  - 两侧至少要有一侧已存在
  - 初次同步时，如果一侧不存在，会用另一侧直接补齐
  - 如果两侧都存在，则比较最后修改时间，用较新一侧覆盖较旧一侧
  - 如果时间戳相同但内容不同，默认保留源路径并覆盖目标路径
  - 在至少完成过一次成功同步后，如果原本两侧都有的文件只剩下一侧，会把仍存在的一侧移入 `.back` 后删除，避免被自动复活
- 文件夹规则
  - 源必须是目录，目标必须是目录路径
  - 首次同步会自动创建目标目录
  - 仅遍历源目录中的文件并复制到目标目录的相对路径上
  - 会删除目标目录中“源目录里已经不存在”的多余文件
  - 删除前会先把这些多余文件移动到目标同级 `.back` 目录
- 双向文件夹规则
  - 两侧至少要有一侧目录已存在
  - 会遍历两侧文件集合和“上次成功同步清单”的并集
  - 新增文件只存在于一侧时，会复制到另一侧
  - 已经成功同步过、但随后只剩一侧存在的文件，会把仍存在的一侧移入 `.back` 后删除
  - 两侧都存在的同名文件，会按最后修改时间决定覆盖方向
  - 仍然不做冲突合并
- 自动同步触发
  - 文件监听：文件规则监听源文件所在目录；目录规则递归监听整个源目录
  - 防抖：监听事件会合并，约 1.5 秒后触发一次实际同步
  - 轮询兜底：后台每 10 秒检查一次规则，到达各自轮询间隔后触发同步
  - 如果某条规则的监听初始化失败，该规则会显示为异常，但不会阻止应用启动；轮询兜底仍可继续工作
- 变更判定
  - 先比较文件大小
  - 再比较修改时间
  - 前两者无法确认时，再计算 blake3 哈希判断内容是否一致
  - 双向规则在内容不一致时，实际覆盖方向由最后修改时间决定；复制后会保留较新一侧的修改时间，避免来回抖动
- 过滤策略
  - `includeGlobs` 为空时，目录规则默认包含全部文件
  - `excludeGlobs` 命中后会直接排除
  - 只有“命中 include 且未命中 exclude”的文件才会被同步
- 冲突与删除策略
  - 当前唯一冲突策略是“覆盖并备份”
  - 单向文件夹规则会把目标侧多余文件移入 `.back` 后再删除
  - 双向规则会在“已建立同步基线”后传播删除，避免单侧删除的文件被另一侧自动恢复
- 备份策略
  - 仅在覆盖已有目标文件前创建备份
  - 备份写入目标同级 `.back` 目录
  - 文件规则直接备份文件；目录规则按相对路径写入备份目录
  - 每次同步后会按保留天数清理过期备份
  - 双向同步时，会在“被覆盖的那一侧”各自写入备份，因此源侧和目标侧都可能出现 `.back` 目录
- 安全校验
  - 源路径必须存在且必须使用绝对路径
  - 文件规则要求源是文件；目录规则要求源是目录
  - 目录规则下，源目录和目标目录不能互相包含
